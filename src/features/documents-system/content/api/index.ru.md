# API-доступ

Шаблон включает доступ для интеграций через API-ключи Better Auth и стартовую поверхность
`/api/v1`. API-клиенты авторизуются заголовком `x-api-key`. Cookies браузерной сессии не
принимаются для внешних API-маршрутов.

## Семейства ключей

| Тип ключа | Где управляется | Субъект доступа | Типичный сценарий |
| --------- | --------------- | --------- | ----------------- |
| Персональный API-ключ | `/user/api-keys` | Пользователь-владелец | Скрипты и интеграции, которые должны следовать участию пользователя в рабочих пространствах. |
| API-ключ рабочего пространства | `/w/:organizationKey/settings/api-keys` | Одна организация рабочего пространства | Серверные интеграции, принадлежащие рабочему пространству. |

Оба семейства используют один внешний заголовок и единые правила формата ответа.

## Стартовые API-маршруты

Начальные маршруты `/api/v1` отдают данные организаций только для чтения:

- `GET /api/v1/me`
- `GET /api/v1/organizations`
- `GET /api/v1/organizations/:organizationId`
- `GET /api/v1/organizations/:organizationId/members`
- `GET /api/v1/organizations/:organizationId/teams`
- `GET /api/v1/organizations/:organizationId/teams/:teamId/members`

## Модель безопасности

Перед возвратом данных проверяются права API-ключа, правила области организации, срок действия и
ограничения частоты запросов. Продуктовые API-маршруты должны расширять эти проверки, а не читать
сырые данные Better Auth в коде маршрута.
