# Рабочее пространство

Workspace — это пользовательское название рабочей области приложения. Внутри шаблона workspace
использует Better Auth Organizations: организация хранит участников, роли, команды, приглашения,
API-ключи и настройки доступа.

Эта страница помогает понять базовые сценарии, которые уже есть в шаблоне и которые можно
расширять при разработке собственного продукта.

## Модель workspace

- **Workspace backed by organization**: пользователь видит workspace, а приложение хранит его как
  Better Auth Organization.
- **Workspace URL**: защищенные рабочие страницы открываются по маршрутам
  `/w/:organizationKey/...`, где `organizationKey` — slug или id организации.
- **Dashboard entry point**: глобальный `/dashboard` перенаправляет пользователя в активный
  workspace. Если активного workspace нет, выбирается стабильный доступный fallback.
- **URL context**: при открытии deep link приложение использует workspace из URL и не меняет
  активный workspace без явного выбора пользователя.

## Настройки workspace

Раздел настроек workspace построен как набор отдельных страниц в общем settings shell.

| Раздел | Что доступно |
| ------ | ------------ |
| Details | Название и slug workspace. Пользователи без права изменения видят значения в read-only режиме. |
| Users | Список участников, текущий пользователь, роли, добавление существующих пользователей и предупреждения по email-domain policy. |
| Invitations | Создание, список, статусы и ссылки приглашений. Раздел виден пользователям с нужными правами. |
| Teams | Создание, переименование, удаление команд и управление участниками команд. Regular members видят команды без управляющих действий. |
| Roles | Зарезервированный раздел для будущего расширения ролей. |
| API keys | Workspace API-ключи для машинного доступа к `/api/v1`, если роль пользователя разрешает управление ключами. |

## Участники и роли

В шаблоне есть встроенные роли **owner**, **admin** и **member**. Owner и admin могут управлять
большинством workspace-настроек, а member сохраняет доступ к чтению там, где это безопасно:
например, к списку участников, командам и контексту настроек.

Удаление workspace защищено от опасных сценариев: пользователь не должен удалить последний
доступный workspace, если после этого он останется без рабочей области.

## Приглашения и email-домены

Workspace admins могут создавать приглашения и копировать ссылку для invitee. После входа invitee
видит страницу решения по приглашению и может принять или отклонить его. Принятие возможно только
для подтвержденного основного email, который совпадает с адресом приглашения.

Workspace может ограничить новые приглашения списком разрешенных email-доменов. Ограничение не
удаляет уже существующих участников, но показывает предупреждения для пользователей вне текущей
политики.

## Команды

Teams — это явные подгруппы внутри workspace. Новый workspace не создает team автоматически: ноль
команд считается нормальным состоянием. Authorized members могут создать команду, переименовать ее,
удалить и добавить в нее существующих участников workspace.

Team-targeted invitation добавляет пользователя в workspace и в выбранную team после принятия
приглашения.

## API-доступ workspace

Workspace API-ключи управляются в настройках workspace и действуют от имени одной organization.
Они подходят для серверных интеграций, которым нужен доступ к read-only `/api/v1` данным
workspace, участников и команд.

Персональные API-ключи живут в настройках аккаунта пользователя и действуют от имени владельца.
Оба типа ключей используют заголовок `x-api-key`, scopes, expiration и rate limits.

## Когда пользователь без workspace

Если у пользователя нет доступных workspace, приложение показывает onboarding guard. Он не
блокирует глобальные страницы аккаунта и страницу управления workspaces: пользователь может создать
workspace или перейти к личным приглашениям.
