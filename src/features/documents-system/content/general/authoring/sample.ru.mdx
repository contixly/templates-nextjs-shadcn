# Возможности документации

Эта страница показывает, как выглядит обычный Markdown и специальные компоненты, доступные авторам документации.
Используйте ее как визуальный пример перед созданием новой страницы.

## Базовый Markdown

### Тексты

Документация поддерживает обычный текст, **акценты**, ~~устаревшие фрагменты~~, ссылки и списки.
Например, можно сослаться на [главную страницу документации](/docs) или открыть
[внешний ресурс](https://nextjs.org/docs) в новой вкладке. Сложные термины и оговорки удобно выносить
в сноски[^gfm], чтобы не перегружать абзац[^style].

- Маркированный список подходит для короткого перечисления.
- Каждый пункт должен быть самостоятельным и понятным.
- Если список становится длинным, лучше разделить его на несколько блоков.

1. Нумерованный список подходит для последовательности действий.
2. Каждый шаг должен описывать одно действие.
3. Результат шага лучше формулировать явно.

> Цитата подходит для принципа, договоренности или выдержки из пользовательского сценария.

Короткие значения можно писать внутри строки: `status: "published"`, `group: "Общее"` или `parentItem: "Авторам"`.

### Блок кода

```md
---
title: "Название страницы"
description: "Короткое описание страницы"
group: "Общее"
parentItem: "Авторам"
order: 30
status: "draft"
toc: true
---
```

### Таблицы

| Поле          | Тип    | Назначение                                                  |
| ------------- | ------ | ----------------------------------------------------------- |
| `title`       | строка | Название страницы, отображается в шапке и левом меню        |
| `description` | строка | Подзаголовок страницы и описание для поиска                 |
| `order`       | число  | Позиция страницы внутри `parentItem`                        |
| `status`      | строка | Статус страницы: `draft`, `review`, `published`, `archived` |
| `hide`        | булево | Скрывает страницу в проде при любом статусе                 |
| `toc`         | булево | Включает правую колонку с навигацией по разделам страницы   |

### Чек-листы

Чек-листы удобно использовать для приёмки страницы перед публикацией:

- [x] Заполнить frontmatter и сверить значения `group` и `parentItem`
- [x] Проверить, что заголовки второго уровня попали в правую навигацию
- [ ] Согласовать описание со стейкхолдером
- [ ] Перевести статус в публикацию — `status: "published"`

## Выделенные заметки

<Callout title="Обычная заметка">
  Такой блок подходит для важного пояснения, которое должно быть заметнее обычного абзаца.
</Callout>

<Callout title="Полезный контекст" variant="info">
  Дополнительная информация, которая помогает понять сценарий, но не блокирует основной поток.
  Используйте, чтобы сослаться на смежные документы или поделиться нюансом.
</Callout>

<Callout title="Готово" variant="success">
  Подтверждение успешного завершения шага: например, страница опубликована, материалы согласованы,
  изменения отражены в навигации.
</Callout>

<Callout title="Предупреждение" variant="warning">
  Используйте предупреждение, когда действие требует внимания: например, перед публикацией страницы
  или изменением статуса материала.
</Callout>

<Callout title="Критичное ограничение" variant="danger">
  Такой вариант подходит для ошибок, блокирующих сценарий, или правил, нарушение которых приводит к
  потере данных.
</Callout>

## Пошаговый сценарий

<Steps>
  <Step title="Подготовьте структуру">
    Определите раздел, родительский пункт меню и порядок страницы в навигации.
  </Step>

<Step title="Заполните содержание">
  Начните с краткого описания, затем добавьте основные разделы, примеры и ограничения.
</Step>

  <Step title="Проверьте результат">
    Убедитесь, что заголовки попали в правую навигацию, а страница отображается в левом меню.
  </Step>
</Steps>

## Рекомендации авторам

Пишите страницу вокруг одного пользовательского сценария. Если материал начинает смешивать роли, статусы, инструкции и
справочную информацию, разделите его на несколько страниц и свяжите их ссылками.

Основные разделы начинайте с заголовка второго уровня. По ним строится правая навигация, поэтому названия должны быть
короткими и узнаваемыми.

[^gfm]: GitHub Flavored Markdown добавляет к базовому Markdown таблицы, чек-листы, зачёркивания, автоссылки и сноски.

[^style]: Сноска формируется через `[^id]` в тексте и `[^id]: текст` где-нибудь в документе — порядок не важен.
