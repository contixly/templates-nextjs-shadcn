# Архитектура функциональных срезов

Используйте функциональный срез для каждой продуктовой возможности. Цель - держать бизнес-поведение
рядом с функцией-владельцем и оставлять файлы маршрутов тонкими.

## Стандартная структура файлов

Код функции живет в `src/features/{feature-name}`:

| Файл или папка | Назначение |
| -------------- | ---------- |
| `actions/` | Серверные действия для мутаций. |
| `components/` | UI-компоненты конкретной функции. |
| `{feature}-repository.ts` | Доступ к данным и кешированные чтения. |
| `{feature}-types.ts` | Типы, DTO и вспомогательные функции тегов кеша. |
| `{feature}-schemas.ts` | Zod-схемы валидации. |
| `{feature}-routes.ts` | Описания маршрутов, принадлежащие функции. |
| `{feature}-logger.ts` | Дочерний структурированный логгер для функции. |

`src/app` должен содержать только страницы, макеты и обработчики маршрутов, которые делегируют в
модули функций.

## Направление зависимостей

Функциональные срезы могут зависеть от общих библиотек и общего UI. Они не должны напрямую
зависеть от других функций. Если двум функциям нужно общее поведение, вынесите его в `src/lib`,
`src/server` или границу общего компонента.

## Сообщения и метаданные

Пользовательский текст живет в `src/messages` и добавляется для обеих поддерживаемых локалей.
Метаданные маршрутов должны использовать существующие вспомогательные функции метаданных страниц, а
не строки, жестко заданные в файлах страниц.

## Публичная документация

Когда функция меняет видимое пользователю поведение, обновите или добавьте соответствующую страницу
публичной документации. Используйте [OpenSpec, E2E и документация](/docs/developers/openspec-e2e-docs) как
чек-лист для связанных поверхностей.
