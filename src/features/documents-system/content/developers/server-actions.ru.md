# Серверные действия

Все продуктовые мутации должны использовать серверные действия, кроме поверхностей фреймворка или
внешнего API. Так валидация, авторизация, мутация, обновление кеша и форма результата остаются
согласованными.

## Последовательность действия

Каждое действие должно идти по одной последовательности:

1. Проверить ввод через Zod.
2. Загрузить аутентифицированного пользователя на сервере.
3. Проверить доступ к целевой сущности.
4. Изменить данные через Prisma.
5. Обновить теги кеша и затронутые пути.
6. Вернуть `ActionResult`.

## Вспомогательные функции защиты

Используйте общие вспомогательные функции защищенных действий вместо повторной реализации
авторизации и обработки ошибок в каждой функции:

- `createProtectedActionWithInput` для действий с проверенным вводом;
- `createProtectedAction` для действий без ввода.

## Обновление кеша

Серверные действия должны использовать вспомогательные функции тегов кеша из файла типов функции.
Обработчики маршрутов, которые меняют данные, должны повторно валидировать затронутые пути.

## Обработка ошибок

Возвращайте безопасные для пользователя сообщения об ошибках и стабильные коды ошибок. Детали
реализации логируйте через логгер функции и не показывайте их в UI.
