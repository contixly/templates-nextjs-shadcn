# Добавить маршрут API v1

Используйте этот процесс, когда продукту нужен новый машиночитаемый маршрут под `/api/v1`.

## Выберите маршрут

Добавьте обработчик маршрута в `src/app/api/v1`. Обработчик должен быть тонким: разобрать входные
данные запроса, потребовать API-ключ, вызвать модули функции или сервера и вернуть стабильный
формат ответа.

## Опишите права

Добавьте конкретное право для нового маршрута. Если несколько маршрутов относятся к одной области,
добавьте или расширьте набор прав, чтобы пользователи могли настраивать ключи без ручного выбора
каждой области доступа.

## Требуйте аутентификацию по API-ключу

Используйте вспомогательные функции аутентификации по API-ключу. Браузерные сессии не должны
давать доступ к `/api/v1`; вызывающая сторона обязана отправить `x-api-key`.

## Возвращайте стабильные ответы

Успешные ответы должны возвращать `{ "data": ... }`. Обработанные ошибки должны возвращать
`{ "error": { "code": "...", "message": "..." } }` с подходящим статусом.

## Документируйте и тестируйте

Обновите [справочник API v1](/docs/api/api-v1), добавьте или обновите требования OpenSpec и
покройте маршрут через API-ориентированные E2E или интеграционные тесты.
