# v0.0.6

Better Auth Teams release for the Next.js application template.

This release builds on `v0.0.5` by turning the previous Teams settings placeholder into a working Better Auth Teams
management surface, adding team-aware invitations, and persisting active team state in sessions.

## Highlights

- Enabled Better Auth Teams on the server and client organization plugins
- Added Prisma persistence for teams, team members, invitation team targets, and active team sessions
- Replaced the workspace Teams settings placeholder with team creation, rename, deletion, member assignment, and active
  team controls
- Added optional team targeting to workspace invitations and invitation review surfaces
- Kept workspaces as the all-members context, with no automatic default team and a valid zero-team state
- Added cache tags, repositories, schemas, localized copy, and tests for team management flows
- Replaced the low-level `ButtonLoading` spinner helper with a reusable `LoadingButton` component
- Reworked account and workspace form validation to use compact inline helper/error rows and form-level error notices

## Included in this release

### Better Auth Teams foundation

- Better Auth Teams are enabled for organization auth on both server and client
- New workspaces do not create an automatic default team
- The workspace organization remains the implicit all-members context
- Users can set or clear `session.activeTeamId` without changing the workspace URL context
- Deleting a team clears the active team before removal and leaves workspace membership intact
- Better Auth session schema now persists `activeTeamId`

### Database and persistence

- Added `teams` and `team_members` tables through Prisma migration `20260428101608_better_auth_teams`
- Added nullable `teamId` support to invitations
- Added nullable `activeTeamId` support to sessions
- Added foreign keys, indexes, and cascade behavior for team and team-member records
- Added a raw SQL unique index for case-insensitive team-name uniqueness within a workspace organization
- Added unique team membership enforcement for each team/user pair

### Workspace team management

- Added a real `/w/:organizationKey/settings/teams` management page in the shared settings shell
- Authorized users can create, rename, and delete explicit workspace teams
- Authorized users can add existing workspace members to teams and remove users from teams
- Team membership changes do not change workspace membership or workspace roles
- Regular members can view teams in read-only mode when they lack management permissions
- Team cards show member counts, member tables, workspace role badges, empty states, and permission-gated controls
- Team names are trimmed, capped at 50 characters, and limited to letters, numbers, spaces, hyphens, and underscores
- Duplicate team names are rejected case-insensitively inside the same workspace

### Team-aware invitations

- Workspace invitation creation can optionally target one team from the current workspace
- Invitation creation rejects invalid or cross-workspace team targets
- Invitation tables show the target team or a workspace-only state
- Invitation decision pages and pending invitation cards display the target team when present
- Accepting a team-targeted invitation grants workspace membership and joins the invited user to the targeted team
- Invitation acceptance now invalidates team-related cache tags in addition to workspace and member cache tags

### Repositories, actions, and cache invalidation

- Added workspace team repository functions using Cache Components with `cacheLife("hours")`
- Added team cache tags for team lists, individual teams, and team members
- Added server actions for creating, updating, deleting, activating, and clearing workspace teams
- Added server actions for adding and removing team members
- Added shared team mutation error handling for duplicate, not-found, permission, and cross-workspace cases
- Team mutations update organization, member, user, team, and team-member cache tags

### UI and localization

- Added English and Russian copy for the Teams settings page, validation errors, invitation team targets, and active team
  state
- Added compact `FieldMessage` rows so field hints and validation errors share the same reserved line instead of
  stretching form spacing
- Added inline `FormErrorNotice` blocks for actionable submit/server errors in account and workspace forms
- Updated the project README to describe Better Auth Teams, team management, and optional team-targeted invitations
- Updated workspace settings navigation specs so Teams is now an implemented section while Roles remains a placeholder
- Reworked custom loading-button behavior into `LoadingButton`, which preserves button footprint while overlaying a
  spinner and setting `aria-busy`

## Upgrade notes

Compared with `v0.0.5`, template users should note:

- A Prisma migration is required for `teams`, `team_members`, `invitations.teamId`, and `sessions.activeTeamId`
- Existing workspaces start with zero explicit teams; this is intentional and valid
- The workspace organization remains the all-members context even when explicit teams exist
- Team names are unique case-insensitively per workspace organization
- Invitations can now carry an optional team target; deleting the team later leaves the invitation as workspace-only
- Active team state is stored on the Better Auth session and can be cleared independently from the active workspace
- Custom loading states should use `LoadingButton` from `src/components/ui/custom/button-loading`
- Form validation UI should use `FieldMessage` and `FormErrorNotice` from `src/components/ui/custom`
