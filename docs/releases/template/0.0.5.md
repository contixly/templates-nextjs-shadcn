# v0.0.5

Workspace email-domain restriction release for the Next.js application template.

This release builds on `v0.0.4` by adding workspace-level allowed email-domain policies for invitations and member
management, stronger visibility for existing members outside active policies, and a broader shadcn/ui component
foundation.

## Highlights

- Added workspace settings for allowed email domains, stored in organization metadata
- Enforced allowed-domain restrictions when creating workspace invitations
- Blocked invitation acceptance when an existing invitation no longer matches the workspace's active domain policy
- Added direct member-add warnings with explicit override for users outside active allowed domains
- Added users-page alerts and badges for existing members outside the active domain policy
- Added domain normalization, validation, exact-domain matching, and localized error copy
- Added new shadcn/ui components and a custom `ButtonLoading` helper for stable button loading states
- Documented that generated shadcn base components in `src/components/ui/` should not be edited directly

## Included in this release

### Workspace allowed email domains

- Workspace settings now include an "Allowed Email Domains" field
- Domain values can be entered one per line or comma-separated
- Domain values are trimmed, lowercased, deduplicated, and may be entered with a leading `@`
- Invalid domain values are rejected before mutating workspace data
- Empty domain settings disable restrictions and allow any email domain
- Allowed domains are persisted in organization metadata without removing unrelated metadata
- Member cache tags are invalidated when the active domain policy changes

### Invitation restrictions

- Invitation creation rejects recipients whose email domain is outside the workspace's allowed-domain policy
- Invitation forms show the active allowed-domain policy when restrictions are configured
- Invitation acceptance now re-checks the workspace's current domain policy
- Pending invitations that become out of policy are shown as domain-restricted and cannot be accepted
- Invitation domain checks run server-side in addition to form-level validation

### Direct member add and member visibility

- Directly adding an existing user outside the active policy now returns a warning instead of immediately adding them
- Workspace admins can explicitly confirm the warning to add an out-of-policy existing user
- Existing workspace memberships are not removed when a policy changes
- Workspace users page now displays a page-level warning when current members are outside active restrictions
- Current user and member rows show an "outside domain policy" badge when applicable
- Workspace member DTOs now include the resolved email domain and policy status

### UI components and loading states

- Added the shadcn `Alert` component and used it for the users-page domain-policy warning
- Added shadcn components for accordion, aspect ratio, button group, calendar, hover card, menubar, pagination, progress,
  radio group, resizable panels, slider, and switch
- Added `react-day-picker` and `react-resizable-panels` dependencies required by the new shadcn components
- Added a custom `ButtonLoading` component under `src/components/ui/custom/`
- Replaced inline button spinners in account and workspace dialogs with `ButtonLoading` to preserve button width while
  loading
- Added repository guidance to keep generated shadcn base components immutable and compose overrides under
  `src/components/ui/custom/`

### OpenSpec and documentation

- Added and archived the OpenSpec change for workspace domain invitation restrictions
- Updated workspace invitation, organization management, and user management specs
- Added English and Russian localized copy for domain policy settings, validation errors, warnings, and invitation states

## Quality and verification

- Added tests for allowed-domain normalization, validation, exact-domain matching, metadata parsing, and metadata merging
- Added action tests for invitation creation with allowed and restricted domains
- Added action tests for direct member-add warnings, acknowledged overrides, and missing target users
- Added tests for invitation acceptance when the decision loader marks an invitation as domain-restricted
- Added workspace settings tests for storing and clearing allowed domains
- Added users-page and loader coverage for out-of-policy member warnings and badges
- Added invitation decision page coverage for the domain-restricted state
- Added `ButtonLoading` tests to verify stable idle and loading states

## Upgrade notes

Compared with `v0.0.4`, template users should note:

- No Prisma migration is required for allowed-domain restrictions; the policy is stored in organization metadata
- Allowed-domain matching is exact, so `example.com` does not automatically allow `team.example.com`
- Existing workspace members are not removed when restrictions are enabled or changed
- Existing pending invitations can become unacceptably domain-restricted if the workspace policy changes after creation
- Invitation creation and acceptance now enforce the active workspace domain policy server-side
- Direct member add can still add out-of-policy existing users, but only after an explicit warning confirmation
- New shadcn components add `react-day-picker` and `react-resizable-panels` to the dependency tree
- Custom behavior for generated shadcn components should live in `src/components/ui/custom/`
