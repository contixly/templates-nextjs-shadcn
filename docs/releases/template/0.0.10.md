# v0.0.10

E2E coverage and automation reliability release for the Next.js application template.

This release builds on `v0.0.9` by expanding OpenSpec-backed Playwright coverage across auth,
workspace, settings, invitation, API key, and `/api/v1` flows. It also tightens local automation
auth, session-cookie cache handling, and reused-server E2E behavior so browser automation runs are
more repeatable.

## Highlights

- Expanded Playwright coverage for account settings, account sessions, workspace management,
  workspace invitations, workspace teams, workspace users, workspace routing, workspace fallbacks,
  API key management, `/api/v1` API key access, local automation auth, and security headers
- Added shared E2E helpers for workspaces, invitations, browser contexts, API keys, local auth,
  route constants, and cold-route retry handling
- Made Playwright run serially by default to reduce cross-scenario state conflicts
- Added explicit `AUTH_DISABLE_SESSION_COOKIE_CACHE` handling for local automation and reused-server
  Playwright runs
- Centralized API key configuration constants in `@lib/api-key-config` while preserving feature-type
  re-exports for existing imports
- Improved workspace creation and settings navigation by revalidating key workspace paths and moving
  slug or route-key changes to the canonical settings route
- Expanded OpenSpec requirements for account, local automation, API key, and workspace fallback
  behavior

## Upgrade notes

Compared with `v0.0.9`, template users should note:

- No Prisma migration is required for this release.
- The template package metadata now reports `0.0.10`; this release does not include dependency
  upgrades.
- Playwright now runs serially by default with `workers: 1`.
- The default `npm run e2e` path starts its own development server and sets the local automation and
  session-cache environment needed by the suite.
- Reused-server runs with `PLAYWRIGHT_START_SERVER=false` require
  `AUTH_DISABLE_SESSION_COOKIE_CACHE=true` on both the already-running app and the Playwright command.
  Global setup now rejects reused-server runs that miss this setting.
- `LOCAL_AUTOMATION_AUTH_ENABLED` remains local-only and must not be enabled in production.
- API key constants now live in `@lib/api-key-config`; they are still re-exported from feature types
  for compatibility with existing imports.
- Workspace slug or route-key changes now navigate to canonical settings routes, so browser tests and
  deep links should expect the updated settings URL after a successful save.
