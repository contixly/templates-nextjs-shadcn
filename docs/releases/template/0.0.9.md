# v0.0.9

API key management and scoped API access release for the Next.js application template.

This release builds on `v0.0.8` by adding Better Auth API keys for personal and organization
machine access, introducing `/api/v1` starter endpoints, and adding settings UI for creating,
updating, disabling, and deleting API keys.

## Highlights

- Added Better Auth API key support with separate personal and organization key families
- Added the Prisma `ApiKey` model and migration for Better Auth key storage
- Added personal API key management at `/user/api-keys`
- Added organization API key management at `/w/:organizationKey/settings/api-keys`
- Added one-time secret display, permission presets, expiration controls, and rate-limit controls
- Added scoped `/api/v1` read-only endpoints authenticated by the `x-api-key` header
- Added an explicit API key principal model for user-owned keys and organization-owned keys
- Added organization access-control permissions for API key create, read, update, and delete actions
- Added API key cleanup when users or organizations are deleted
- Added OpenSpec requirements and Playwright coverage for API key workflows

## Upgrade notes

Compared with `v0.0.8`, template users should note:

- A Prisma migration is required. Apply `prisma/migrations/20260627171629_add_api_key_table/migration.sql` before using
  API key features.
- Regenerate the Prisma client after installing dependencies so `prisma.apiKey` is available.
- API clients must send API keys in the `x-api-key` header; browser sessions are not accepted by `/api/v1` routes.
- Personal keys act as the owning user and lose organization access when that user is no longer a member.
- Organization keys act as organization principals and are scoped to the owning organization until disabled, expired,
  deleted, rate-limited, or missing required permissions.
- Owner and admin organization roles can manage organization API keys by default; member roles cannot.
- API key secrets are shown only once at creation time. List and update flows expose only non-secret metadata.
- API key expiration is not renewed by a normal edit save. Users must explicitly enable renewal in the edit dialog.
- Existing custom permission records are preserved unless the user explicitly replaces scopes through presets.
- `/api/v1` returns `{ data: ... }` for successful responses and `{ error: { code, message } }` for handled failures.
- The release notes directory for the template is now `docs/releases/template/`.
- Node.js 22+ is recommended for local development and deployment because the current cache handler dependency requires
  it.
