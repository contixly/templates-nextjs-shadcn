# v0.0.4

Security hardening release for the Next.js application template.

This release builds on `v0.0.3` by tightening runtime security defaults, authentication redirect handling, protected
action boundaries, account session management, and workspace invitation privacy.

## Highlights

- Added application-wide security response headers, including frame, content type, referrer, permissions, CSP, and
  production HSTS defaults
- Replaced placeholder base URL handling with normalized `NEXT_PUBLIC_APP_BASE_URL` and `BETTER_AUTH_URL` configuration
- Hardened auth redirects against protocol-relative, traversal, and backslash-normalized external targets
- Stopped trusting middleware-forwarded user id headers as the protected action authorization boundary
- Removed Better Auth session tokens from client-facing session management data
- Prevented workspace invitation metadata disclosure when the signed-in user is not the invited recipient
- Added database-level protection against duplicate pending invitations for the same workspace and email
- Validated workspace slug uniqueness across all organizations, including organizations the actor cannot access

## Included in this release

### Runtime security headers

- Global `next.config.ts` headers now apply the template's security defaults to every route
- `X-Content-Type-Options: nosniff` prevents MIME sniffing
- `X-Frame-Options: DENY` and `Content-Security-Policy` `frame-ancestors 'none'` block framing
- `Referrer-Policy: strict-origin-when-cross-origin` limits referrer leakage
- `Permissions-Policy` disables camera, microphone, and geolocation by default
- Production responses include `Strict-Transport-Security` with subdomain coverage
- Remote image loading no longer allows arbitrary HTTPS hosts by default

### Base URL and auth origin handling

- `APP_BASE_URL` is now derived from `NEXT_PUBLIC_APP_BASE_URL` or `BETTER_AUTH_URL`
- Production startup fails fast when no app base URL is configured
- `APP_BASE_DOMAIN` is derived from the normalized app URL instead of a placeholder domain
- Better Auth allowed hosts and trusted origins are built from the configured app and auth URLs
- The auth client now uses the shared normalized app base URL
- README, `.env.example`, template setup notes, and Open Graph branding were updated for the new variable name

### Redirect and route safety

- `sanitizeRedirectPath` now rejects protocol-relative redirects such as `//example.com`
- Redirect sanitization rejects paths that become protocol-relative after traversal cleanup
- Redirect sanitization rejects backslashes before WHATWG URL normalization can turn them into external targets
- OAuth sign-in callback and new-user redirect URLs now pass through the same sanitizer
- Regression coverage was added for valid in-app paths and unsafe redirect variants

### Protected actions and account sessions

- Protected action helpers now load the current user id from a validated Better Auth session
- Middleware no longer injects `x-user-id` into forwarded request headers
- Protected action context now carries request headers so Better Auth mutations use the current request context
- Client-facing session lists now include session ids and display metadata, not raw Better Auth session tokens
- Single-session revocation resolves the token server-side after verifying ownership
- The single-session revocation action refuses to revoke the current session through the other-session flow

### Workspace invitations and membership safety

- Invitation decision pages now hide invitation details when the signed-in user's email does not match the invitation
- Accepting an invitation now also sets the accepted workspace as the active organization
- Invitation creation normalizes email addresses before duplicate and membership checks
- Expired pending invitations for the same workspace and email are canceled before a new invitation is created
- A partial unique database index prevents concurrent duplicate pending invitations per workspace and normalized email
- Duplicate invitation errors now return the invitation-specific conflict message
- Member-add and invitation flows pass request headers through Better Auth organization APIs
- Role selection forms now disable submission when no assignable roles are available

### Workspace identity safeguards

- Workspace slug updates now check for collisions against all organizations, not only organizations visible to the actor
- Workspace renames preserve the current slug unless a new slug is explicitly submitted
- Workspace creation retries slug generation on unique-constraint races before returning a duplicate-slug conflict
- Repository helpers now expose direct organization lookup by slug for global collision checks

## Quality and verification

- Added tests for redirect sanitization edge cases
- Added tests for server-side session-token resolution and current-session revoke protection
- Added tests for invitation recipient mismatch privacy
- Added tests for accepting invitations and updating the active organization
- Added tests for global workspace slug collision handling
- Added coverage for invitation page intro rendering and session list data shape changes

## Upgrade notes

Compared with `v0.0.3`, template users should note:

- Replace `PUBLIC_BASE_URL` with `NEXT_PUBLIC_APP_BASE_URL` in all environments
- Production deployments must configure `NEXT_PUBLIC_APP_BASE_URL` or startup will fail
- Existing deployments should run the new Prisma migration before using invitation creation flows
- The pending invitation unique index can fail to apply if duplicate pending invitations already exist; resolve
  duplicates
  before migrating production data
- Remote images are no longer allowed from every HTTPS host; add explicit `images.remotePatterns` entries if your app
  depends on external image domains
- Auth redirect targets are now constrained to in-app relative paths
- Workspace name changes no longer regenerate slugs automatically; submit `slug` when an explicit slug change is needed
