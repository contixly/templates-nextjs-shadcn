# v0.0.7

Distributed caching release for the Next.js application template.

This release builds on `v0.0.6` by adding configurable Redis/Valkey-backed cache handlers for Cache Components and ISR,
improving workspace route streaming boundaries, and hiding social login providers that are not configured in the
environment.

## Highlights

- Added optional Redis/Valkey-backed cache handlers for Next.js Cache Components and ISR
- Disabled the built-in in-memory Next.js cache when the custom handlers are configured
- Added shared cache settings for remote cache enablement, Redis connection values, and key prefixes
- Added workspace and dashboard Suspense boundaries with route-shaped skeleton fallbacks
- Removed the full-route workspace loading fallback in favor of local streaming boundaries
- Enabled Better Auth social providers only when their required environment variables are configured
- Updated login, navigation, and account connection UI to render only configured social providers
- Added regression tests for cache configuration, streaming boundaries, and provider filtering

## Included in this release

### Distributed Next.js caching

- Added `@mrjasonroy/cache-components-cache-handler` and `ioredis`
- Configured `cacheHandlers.default` and `cacheHandlers.remote` for Cache Components data caching
- Configured `cacheHandler` for the Next.js ISR/server cache path
- Added a local memory fallback when remote caching is disabled
- Added a Redis/Valkey-backed handler when `REMOTE_CACHING_ENABLED` is enabled
- Added shared cache key and tag prefixes through `REMOTE_CACHING_PREFIX`
- Added support for `REDIS_URL` or `VALKEY_URL`, with optional `REDIS_PASSWORD`
- Added a startup configuration assertion so remote caching fails fast when enabled without a Redis/Valkey URL
- Kept `updateTag`, `revalidateTag`, and `revalidatePath` helpers server-only

### Workspace streaming and skeleton states

- Wrapped the organization dashboard in a local Suspense boundary with a dashboard-shaped skeleton
- Split async workspace settings page content from immediately rendered route intros
- Added skeleton states for workspace, users, teams, invitations, roles, and settings navigation
- Kept settings page children visible while the workspace settings navigation loads
- Converted redirect-only workspace routes to local empty Suspense fallbacks
- Removed the broad `loading.tsx` fallback for `/w/:organizationKey`

### Configured social auth providers

- Added server-only provider discovery based on the required OAuth environment variables
- Better Auth now receives only providers that are fully configured
- Account linking trusted providers now matches the configured provider set
- Yandex generic OAuth is registered only when Yandex credentials are present
- Login, home navigation, site header, and account connections render only configured provider buttons
- Shared social provider metadata remains client-safe and does not read `process.env`

## Quality and verification

- Added infrastructure tests for the Next.js cache handler configuration and cache file layout
- Added auth tests for provider environment filtering and Better Auth provider registration
- Added login form coverage to verify that unconfigured providers are hidden
- Added workspace streaming-boundary tests for dashboard and settings routes
- Updated workspace redirect and settings tests for local Suspense shells
- Updated settings layout tests for navigation skeleton behavior

## Upgrade notes

Compared with `v0.0.6`, template users should note:

- Node.js 22+ is now recommended because the cache handler dependency declares `node >=22`
- No Prisma migration is required for this release
- Remote caching is disabled by default with `REMOTE_CACHING_ENABLED=false`
- To enable distributed caching, configure `REMOTE_CACHING_ENABLED=true` and either `REDIS_URL` or `VALKEY_URL`
- `REDIS_PASSWORD` is optional and is injected into the connection URL only when the URL has no password already
- `REMOTE_CACHING_PREFIX` should be unique per deployed app/environment when sharing a Redis/Valkey instance
- OAuth provider buttons now appear only when the required provider credentials are configured
- The `/w/:organizationKey/loading.tsx` route fallback was removed; workspace routes now use local Suspense boundaries
